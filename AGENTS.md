# 项目规范（Vue3）

本文件用于约束 Codex 在当前项目中的默认行为。  
目标是：小范围修改、代码可运行、风格一致、避免无关扩展。

---

## 技术栈

- Vite + Vue 3 + `<script setup>` + TypeScript
- Vue Router 4.x
- Pinia
- Element Plus
- Axios
- micro-app

---

## 基础代码规范

### Vue 组件

- 使用单文件组件 `.vue`
- 使用 `<script setup lang="ts">`
- 组件名使用 PascalCase
- 优先使用 Composition API
- 组件内部逻辑顺序建议为：
  - props / emits
  - state / computed
  - methods
  - watch / lifecycle

### TypeScript

- 保持类型清晰，避免随意使用 `any`
- 接口类型优先集中定义或就近定义，避免重复散落
- 不新增未使用的类型、变量、函数和 import
- 不生成与 `.vue` / `.ts` 文件同名的无关 `.js` 文件

### Pinia

- 使用 `defineStore`
- 不在组件内堆积复杂业务逻辑
- 状态命名清晰，避免冗余字段
- 不为一次性逻辑强行抽到 store

### Vue Router

- 路由配置集中管理
- 页面组件使用懒加载
- `meta` 用于权限、标题、缓存等页面信息
- 不在无关任务中调整路由结构

### Axios

- 请求统一封装在 `/api`
- 不在组件中直接写 `axios`
- token、错误提示、登录失效等逻辑通过拦截器或统一方法处理
- 接口字段严格按接口文档或用户明确给出的字段使用，不扩散无依据的兼容字段名

### Element Plus

- 优先使用 Element Plus 现有组件
- 不重复封装 Element Plus 已有能力
- 表单使用 `el-form`、`el-form-item`、校验规则等规范写法
- 不随意引入新的 UI 框架或组件库

### micro-app

- 子应用保持独立路由
- 避免污染全局 `window`
- 避免全局样式影响主应用或其他子应用
- 应用间通信优先使用 micro-app 提供的 API

---

## 默认修改原则

- 默认只改当前任务相关文件
- 默认不做架构级调整
- 默认不扩展需求
- 默认不顺手重构
- 默认不修改无关格式
- 默认不删除已有死代码，除非与当前任务直接相关
- 默认不新增依赖；确需新增时，必须先说明原因
- 修改保持小范围、可验证、可回退

---

## Codex 执行规则

- 编码前先明确当前任务目标和必要假设
- 如果需求存在多种理解：
  - 能按最小改动合理推断时，直接执行并说明假设
  - 会影响接口、路由、权限、数据结构、构建配置时，先确认
- 优先选择简单直接的实现
- 不为单次使用代码提前抽象
- 不增加未要求的灵活配置
- 新增的未使用 import、变量、函数必须清理
- 已有无关问题不借机扩展处理
- 多步骤任务按“实现点 + 验证方式”推进
- 完成后说明：
  - 修改了哪些文件
  - 每个文件改了什么
  - 是否已验证
  - 如未验证，说明原因

---

## 输出要求

- 说明变更时，只列出修改点，不粘贴完整文件
- 需要给代码片段时，只返回关键修改部分
- 不返回与任务无关的大段解释
- 代码必须可直接运行
- 不新增无关依赖
- 不输出无关的完整文件内容

---

## 注释规范

- 只在以下场景添加简短注释：
  - 数据来源不明显
  - 状态联动容易误读
  - 解析逻辑较复杂
  - 兼容逻辑容易被误删
- 注释不超过 5 行
- 不为显而易见的代码添加注释
